7. DATABASE & DATA ARCHITECTURE
Schema design for multi-tenant financial data
7.1 Core schema design principles
Multi-tenancy
Row-level security (RLS) in PostgreSQL. Every table has company_id. Tenant isolation
enforced at DB level  not just application level.
Financial precision
All monetary amounts stored as INTEGER (paise)  never FLOAT. GST rates stored as
NUMERIC(5,2). Eliminates floating-point errors.
Audit immutability
Financial vouchers are NEVER updated or deleted. Corrections are new reversal entries.
Full audit trail by design.
Soft deletes
All records use deleted_at timestamp, never hard deletes. GDPR/DPDP right-to-erasure
handled separately via anonymisation.
Event sourcing (Phase
3)
Agent actions emit events to an immutable event log. Current state is derived from events.
Full replay capability.

9. SECURITY, COMPLIANCE & DPDP ARCHITECTURE
Building trust with enterprises that share financial data
9.1 Data security layers
Encryption at rest
AES-256 via AWS KMS. Field-level encryption for PAN, Aadhaar-linked data, bank
account numbers. GSTN credentials stored in AWS Secrets Manager.
Encryption in transit
TLS 1.3 everywhere. Certificate pinning on mobile app. HSTS headers. No HTTP
anywhere in production.
Authentication
Supabase JWT (15-min access tokens). Refresh token rotation. OTP via Firebase for all
logins. MFA mandatory for CA partner accounts.
Authorisation
Row-Level Security in PostgreSQL  tenants cannot access each other's data even with
a SQL injection. Role-based access: owner > CFO > accountant > viewer.
API security
Rate limiting (100 req/min standard). OWASP API Top 10 compliance. Input sanitisation
via Zod. SQL injection impossible via Prisma parameterised queries.
Audit logging
Immutable append-only log of every data access and modification. Includes: user,
timestamp, IP, action, before/after values. Retained 7 years.
Penetration testing
Annual external pentest from Phase 3. Automated OWASP ZAP scans in CI/CD pipeline.
9.2 DPDP Act 2023 compliance checklist
Consent management
Explicit consent collected at signup for each data processing purpose. Granular: billing
data, analytics, credit assessment are separate consents.
Data minimisation
Collect only what's needed for each feature. Aadhaar number never stored  only
Aadhaar e-sign token.
Right to erasure
Customer-initiated deletion triggers: anonymise personal fields, retain financial records as
legally required (7 years), delete analytics data.
Data breach response
72-hour breach notification pipeline. Automated detection via AWS GuardDuty. Incident
response runbook documented.
Data localisation
All data stored in AWS Mumbai (ap-south-1). No cross-border transfer of personal
financial data.
Data Protection Officer
Appoint DPO from Phase 3 (mandatory for financial data processors). DPO contact visible
in product and privacy policy.
Privacy by design
Privacy impact assessment for each new feature before development. Data model
reviewed against DPDP requirements at design stage.

12.4 The non-negotiable first hire
Your most important first hire is not a developer  it is a practising CA with 5+ years of GST and
TDS experience who is willing to work as a technical compliance co-founder or senior advisor.
Every line of compliance logic your CTO writes must be signed off by this person. A software
engineer who 'read the GST Act' is not a substitute. The cost of one wrong tax calculation at scale
is a customer-destroying compliance disaster. The CA's salary is the cheapest insurance you will
ever buy.
